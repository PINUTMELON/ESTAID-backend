# ESTAID 백엔드 개발노트 - 심사 대비 Q&A

> 작성일: 2026-03-27
> 프로젝트: ESTAID - AI 기반 2차 창작 콘텐츠 생성 플랫폼

---

## 1. 프로젝트 개요 관련

### Q: 이 프로젝트가 뭔가요? 한 줄로 설명해주세요.
> **A:** 사용자가 스토리 아이디어를 입력하면 AI가 자동으로 장면(씬)을 분할하고, 각 장면의 이미지와 영상을 생성해서 2차 창작 애니메이션 콘텐츠를 만들어주는 플랫폼입니다.

### Q: 기술 스택을 설명해주세요.
> **A:**
> - **백엔드:** Spring Boot 3.2 / Java 17 / Maven
> - **데이터베이스:** Supabase (PostgreSQL) + Supabase Storage (이미지 저장)
> - **AI 서비스:** Claude API (스토리 생성), FAL.ai FLUX Kontext (이미지 생성), FAL.ai Wan 2.1 (영상 생성)
> - **인증:** JWT 기반 Stateless 인증
> - **비동기 처리:** Spring @Async + Custom ThreadPool

### Q: 왜 이 기술 스택을 선택했나요?
> **A:**
> - **Spring Boot:** 엔터프라이즈 레벨의 안정성, 풍부한 생태계, 비동기 처리(@Async) 지원이 좋아서 AI API 호출처럼 시간이 오래 걸리는 작업에 적합합니다.
> - **Supabase:** PostgreSQL 기반이라 관계형 데이터 모델링에 강하고, Storage 서비스를 함께 제공해서 이미지 업로드까지 하나의 플랫폼에서 처리할 수 있습니다.
> - **Claude API:** 스토리보드 생성에 필요한 창의적 텍스트 생성 능력이 뛰어나고, JSON 포맷으로 구조화된 응답을 안정적으로 생성합니다.
> - **FAL.ai:** FLUX Kontext 모델은 참조 이미지 기반 일관된 캐릭터 생성이 가능하고, Wan 2.1은 시작/끝 프레임 기반 영상 생성을 지원합니다.

## 2. 아키텍처 관련

### Q: 전체 시스템 아키텍처를 설명해주세요.
> **A:** 클라이언트(프론트엔드) → Spring Boot REST API → 3개의 외부 AI 서비스로 구성됩니다.
> 1. 사용자가 스토리 아이디어를 입력하면, **Claude API**가 장면(Scene)을 자동 분할하고 각 장면의 프롬프트를 생성합니다.
> 2. 각 장면의 첫 프레임/마지막 프레임 이미지를 **FAL.ai FLUX Kontext**가 생성합니다.
> 3. 두 프레임 이미지를 기반으로 **FAL.ai Wan 2.1**이 중간 모션을 보간하는 영상을 생성합니다.
> 4. 모든 AI 생성 작업은 **비동기(@Async)**로 처리되며, 프론트엔드는 **폴링(Polling)**으로 완료 여부를 확인합니다.

### Q: 왜 비동기(Async) 처리를 사용했나요?
> **A:** 이미지 생성은 최대 4분, 영상 생성은 최대 10분까지 소요됩니다. 동기 방식이면 클라이언트가 응답을 기다리는 동안 커넥션이 블로킹되어 서버 리소스가 낭비되고, HTTP 타임아웃이 발생합니다. 비동기 처리로 요청 즉시 PENDING 상태를 반환하고, 별도 스레드에서 생성 작업을 수행한 뒤 DB 상태를 업데이트하는 방식을 사용했습니다.

### Q: 비동기 처리의 스레드 풀 설정은 어떻게 했나요?
> **A:** `AsyncConfig`에서 `ThreadPoolTaskExecutor`를 커스텀 설정했습니다.
> - **Core Pool Size: 4** (기본 상시 대기 스레드)
> - **Max Pool Size: 8** (최대 동시 처리 스레드)
> - **Queue Capacity: 50** (대기열 크기)
> - 이미지/영상 생성이 I/O 바운드 작업(외부 API 대기)이라 CPU 코어 수보다 많은 스레드를 할당했고, 메모리 과사용 방지를 위해 최대 8개로 제한했습니다.

### Q: 왜 WebSocket이 아닌 폴링(Polling) 방식을 사용했나요?
> **A:** 구현 복잡도 대비 효용을 고려한 선택입니다. WebSocket은 실시간 양방향 통신에 유리하지만, 연결 관리 및 재연결 로직 등 추가 복잡성이 큽니다. 현재 서비스 규모에서는 몇 초 간격의 폴링으로도 사용자 경험에 큰 차이가 없고, HTTP 기반이라 디버깅과 모니터링도 쉽습니다. 추후 사용자가 늘어나면 SSE(Server-Sent Events)나 WebSocket으로 전환할 수 있습니다.

### Q: JSON을 DB 컬럼에 직접 저장한 이유는?
> **A:** `Plot` 엔티티의 `scenes_json` 컬럼에 Scene 배열을 JSON으로 저장합니다. 이유는:
> 1. **유연한 스키마:** Scene의 필드가 자주 변경될 수 있는데(예: composition enum 추가), JSON이면 별도 마이그레이션 없이 대응 가능합니다.
> 2. **원자적 조회:** 하나의 Plot에 속한 Scene 전체를 한 번의 쿼리로 가져올 수 있어 N+1 문제가 발생하지 않습니다.
> 3. **트레이드오프:** Scene 단위 쿼리(예: "특정 composition인 Scene 검색")는 어렵지만, 현재 비즈니스 요구사항에 그런 쿼리가 없어 문제없습니다.

## 3. AI 서비스 연동 관련

### Q: Claude API를 어떤 용도로 사용하나요?
> **A:** 두 가지 용도로 사용합니다.
> 1. **씬 생성(Scene Generation):** 사용자의 스토리 아이디어를 입력받아 장면별로 분할하고, 각 장면의 구도(composition), 배경, 조명, 스토리, 프레임 프롬프트를 JSON 배열로 생성합니다.
> 2. **영상 프롬프트 생성(Video Prompt):** 첫/끝 프레임 이미지의 프롬프트를 기반으로, 두 프레임 사이의 동작과 카메라 움직임을 묘사하는 영상 생성 프롬프트를 작성합니다.

### Q: Claude API 호출 시 프롬프트 엔지니어링은 어떻게 했나요?
> **A:**
> - **System Prompt**에 역할을 정의합니다: "You are a 2nd creation storyboard writer..."
> - **구조화된 출력을 요구**합니다: JSON 배열 포맷, 필수 필드 명시, composition enum 값 제한(10가지)
> - **캐릭터 정보를 전달**합니다: 캐릭터 이름과 외모 설명을 프롬프트에 포함시켜 일관된 캐릭터 묘사가 나오도록 합니다.
> - **응답 파싱:** Claude가 가끔 마크다운 코드블록(```json)으로 감싸서 응답하는 경우를 처리하는 파싱 로직이 있습니다.

### Q: FLUX Kontext 모델의 특징과 선택 이유는?
> **A:** FLUX Kontext는 **참조 이미지(reference image) 기반 이미지 생성**을 지원합니다. 캐릭터의 참조 이미지 URL을 함께 전달하면, 해당 캐릭터의 외모를 유지하면서 다른 포즈/배경의 이미지를 생성합니다. 이것이 핵심입니다. 일반 text-to-image 모델은 같은 캐릭터라도 매번 다른 얼굴/체형이 나오는데, FLUX Kontext는 시각적 일관성(visual consistency)을 보장합니다.

### Q: Wan 2.1 FLF2V 모델은 뭐가 특별한가요?
> **A:** **FLF2V는 First-Last-Frame to Video의 약자**입니다. 시작 프레임 이미지와 끝 프레임 이미지를 입력하면, 두 이미지 사이의 자연스러운 동작을 보간(interpolation)하여 영상을 생성합니다. 기존의 text-to-video나 image-to-video 모델보다 **결과물의 시작과 끝을 제어할 수 있다는 점**이 장점이며, 스토리보드 기반 콘텐츠 생성에 매우 적합합니다.

### Q: 영상 생성은 왜 비동기 큐 방식인가요?
> **A:** FAL.ai의 Wan 2.1 모델은 영상 생성에 수 분이 걸리기 때문에, FAL.ai 자체적으로 동기 API를 제공하지 않고 **큐 기반 비동기 API**를 제공합니다. 작업을 제출(submit)하면 `request_id`를 받고, 이후 상태를 주기적으로 확인(polling)합니다. 저희 서버에서는 **지수 백오프(exponential backoff)** 전략을 적용해서 초기 2초 → 4초 → 8초 → 최대 15초 간격으로 폴링하여 FAL.ai 서버 부하를 줄였습니다.

### Q: 지수 백오프(Exponential Backoff)가 뭔가요? 왜 사용했나요?
> **A:** 지수 백오프는 재시도 간격을 점점 늘려가는 전략입니다. 처음에는 2초 후 재시도, 다음은 4초, 그 다음은 8초... 이런 식으로 간격을 2배씩 늘립니다. 이유는:
> 1. **서버 부하 감소:** 일정 간격으로 계속 요청하면 서버에 불필요한 부하를 줍니다.
> 2. **네트워크 효율성:** 생성 초기에는 당연히 미완료 상태이므로 자주 확인할 필요가 없습니다.
> 3. **최대 간격 제한(15초):** 너무 오래 기다리면 완료 감지가 늦어지므로 상한선을 둡니다.

### Q: 왜 이미지/영상 AI로 FAL.ai를 선택했나요?
> **A:**
> 1. **모델 선택의 자유도:** FAL.ai는 FLUX, Stable Diffusion, Wan 2.1 등 다양한 최신 오픈소스 모델을 하나의 API로 사용할 수 있는 플랫폼입니다. 직접 GPU 서버를 구축하지 않아도 최신 모델을 바로 쓸 수 있습니다.
> 2. **FLUX Kontext — 캐릭터 일관성:** 저희 서비스의 핵심 요구사항이 "같은 캐릭터가 여러 장면에서 동일한 외모를 유지하는 것"인데, FLUX Kontext 모델이 참조 이미지(reference image) 기반 생성을 지원합니다. Midjourney나 DALL-E에서는 이런 image-to-image 일관성 제어가 어렵습니다.
> 3. **Wan 2.1 FLF2V — 시작/끝 프레임 제어:** 일반적인 text-to-video 모델은 결과물의 끝을 제어할 수 없는데, Wan 2.1 FLF2V는 시작 이미지와 끝 이미지를 모두 입력받아 그 사이를 보간합니다. 스토리보드 기반 콘텐츠 생성에 가장 적합한 모델이었습니다.
> 4. **백엔드 연동에 적합한 API 구조:** 이미지 생성은 동기 API(요청 후 바로 결과), 영상 생성은 큐 기반 비동기 API(제출 후 폴링)로 제공되어, Spring WebClient로 깔끔하게 연동할 수 있었습니다.
> 5. **비용:** Replicate, RunwayML 등 경쟁 서비스 대비 per-request 과금이 저렴하고, 프로토타입 단계에서 부담 없이 테스트할 수 있었습니다.

### Q: Midjourney나 DALL-E는 왜 안 썼나요?
> **A:** Midjourney는 API를 공식 제공하지 않아 백엔드 연동이 불가능합니다. DALL-E는 API가 있지만, 참조 이미지 기반으로 일관된 캐릭터를 생성하는 기능이 부족합니다. 저희에겐 캐릭터 일관성이 핵심 요구사항이라 FLUX Kontext가 필수였고, 이를 API로 제공하는 것이 FAL.ai였습니다.

### Q: 직접 AI 모델을 호스팅하는 건 고려 안 했나요?
> **A:** 고려했지만, GPU 서버 비용(A100 기준 월 수백만 원)과 모델 배포/운영 부담이 큽니다. 모델이 업데이트될 때마다 재배포도 필요합니다. FAL.ai를 쓰면 인프라 관리 없이 최신 모델을 즉시 사용할 수 있어서, 제한된 개발 기간 내에 서비스 로직 개발에 집중할 수 있었습니다.

### Q: AI API 호출 실패 시 어떻게 처리하나요?
> **A:**
> - **이미지 생성:** 타임아웃 시 **1회 자동 재시도** 후 실패 처리합니다. 실패하면 DB 상태를 FAILED로 업데이트하고 에러 메시지를 로깅합니다.
> - **영상 생성:** 10분 타임아웃 내에 완료되지 않으면 FAILED 처리합니다. 폴링 중 에러 응답을 받으면 즉시 실패 처리합니다.
> - **Claude API:** 60초 타임아웃, 실패 시 502 Bad Gateway 에러를 클라이언트에 반환합니다.
> - 모든 실패 케이스에서 **DB 상태는 반드시 업데이트**되어 프론트엔드가 실패를 인지할 수 있습니다.

## 4. 데이터베이스 관련

### Q: 왜 Supabase를 사용했나요?
> **A:**
> 1. **PostgreSQL 기반:** 관계형 데이터 모델링이 가능하고, CHECK 제약조건 등 고급 기능을 사용할 수 있습니다.
> 2. **Storage 통합:** 이미지 파일 업로드/관리를 위한 별도 S3 세팅 없이, Supabase Storage로 즉시 사용 가능합니다.
> 3. **무료 티어:** 프로토타입 단계에서 비용 부담 없이 사용할 수 있습니다.
> 4. **REST API:** 필요 시 프론트엔드에서 직접 접근도 가능합니다.

### Q: DDL을 왜 직접 관리하나요? (ddl-auto: validate)
> **A:** Hibernate의 `ddl-auto: update/create`는 편리하지만, 프로덕션 환경에서는 위험합니다. 인덱스 생성, CHECK 제약조건, 트리거 등 Hibernate가 자동 생성하지 못하는 DDL이 많고, 의도치 않은 스키마 변경이 발생할 수 있습니다. `validate` 모드로 설정하고 `schema.sql`을 직접 관리하면 스키마 변경 이력을 Git으로 추적할 수 있고, 정확히 의도한 대로만 DB가 구성됩니다.

### Q: 엔티티 ID를 왜 UUID(String)로 사용하나요?
> **A:**
> 1. **보안:** Auto-increment ID는 순차적이라 리소스 열거 공격(IDOR)에 취약합니다. UUID는 예측 불가능합니다.
> 2. **분산 환경 대비:** UUID는 서버가 여러 대여도 충돌 없이 ID를 생성할 수 있습니다.
> 3. **트레이드오프:** UUID는 인덱스 크기가 크고 정렬이 안 된다는 단점이 있지만, 현재 규모에서는 문제없습니다.

### Q: 커넥션 풀 설정은 어떻게 했나요?
> **A:** HikariCP를 사용하며:
> - **Core: 5, Max: 15** — 비동기 이미지/영상 생성 스레드(최대 8개)가 동시에 DB를 업데이트할 수 있으므로, 기본 커넥션보다 여유 있게 설정했습니다.
> - **Connection Timeout: 30초** — 커넥션 획득 대기 시간
> - **Idle Timeout: 10분** — 유휴 커넥션 반환 시간
> - **Max Lifetime: 30분** — 커넥션 최대 수명 (Supabase의 커넥션 정리와 맞춤)

### Q: 인덱스는 어떻게 설계했나요?
> **A:** 자주 조회되는 패턴을 기준으로 복합 인덱스를 설계했습니다:
> - **Images:** `(plot_id, scene_number, frame_type)` — 특정 씬의 특정 프레임 이미지를 빠르게 조회
> - **Videos:** `(plot_id, scene_number, video_type, created_at)` — 특정 씬의 최신 영상 조회
> - **Projects:** `(user_id, created_at DESC)` — 사용자별 프로젝트 최신순 목록
> - **Rankings:** `(average_rating DESC, rating_count DESC, created_at)` — 갤러리 인기순 정렬

## 5. 인증/보안 관련

### Q: JWT 인증 방식을 설명해주세요.
> **A:**
> 1. 클라이언트가 `POST /api/auth/login`에 username/password를 전송합니다.
> 2. 서버가 검증 후 JWT 토큰을 생성하여 반환합니다. 토큰에는 `userId`가 포함됩니다.
> 3. 이후 모든 API 요청의 `Authorization: Bearer {token}` 헤더에 토큰을 포함합니다.
> 4. `JwtAuthenticationFilter`가 매 요청마다 토큰을 검증하고, userId를 `SecurityContextHolder`에 설정합니다.
> 5. **Stateless:** 서버에 세션을 저장하지 않아 수평 확장에 유리합니다.

### Q: CORS 설정은 어떻게 했나요?
> **A:** `CorsConfig`에서 허용 오리진을 명시적으로 지정합니다: localhost(개발), 프로덕션 도메인, Cloudflare Tunnel URL. `*`(전체 허용)은 사용하지 않아 보안을 유지합니다. 허용 메서드는 GET, POST, PUT, DELETE, OPTIONS이며, 인증 헤더(Authorization)를 허용합니다.

## 6. API 설계 관련

### Q: REST API 설계 원칙을 설명해주세요.
> **A:**
> - **자원 중심 URL:** `/api/projects`, `/api/plots`, `/api/images`, `/api/videos`
> - **HTTP 메서드:** GET(조회), POST(생성), PUT(수정), DELETE(삭제)
> - **일관된 응답 형식:** `ApiResponse<T>` 래퍼로 `{ success, data, message }` 형태 통일
> - **적절한 상태 코드:** 200(성공), 201(생성), 400(검증실패), 401(인증실패), 404(미존재), 409(충돌), 502(외부 API 에러)

### Q: 에러 처리는 어떻게 하나요?
> **A:** `GlobalExceptionHandler`에서 `@RestControllerAdvice`로 전역 예외를 처리합니다. 커스텀 `BusinessException`에 HTTP 상태 코드와 메시지를 담아 던지면, 핸들러가 일관된 `ApiResponse` 형태로 변환하여 응답합니다. 예상치 못한 예외는 500 Internal Server Error로 처리하되, 스택 트레이스는 클라이언트에 노출하지 않습니다.

## 7. 핵심 비즈니스 로직 관련

### Q: 전체 콘텐츠 생성 파이프라인을 설명해주세요.
> **A:**
> ```
> 1. 프로젝트 생성 → 캐릭터 등록 (참조 이미지 업로드)
> 2. 스토리 아이디어 입력 → Claude API가 장면(Scene) 분할 + 프롬프트 생성
> 3. 각 장면의 첫/끝 프레임 이미지 생성 → FLUX Kontext (참조 이미지 기반)
> 4. Claude가 영상 프롬프트 생성 → Wan 2.1이 시작/끝 프레임 기반 영상 생성
> 5. 갤러리에 공개 → 다른 사용자가 평점(1-5) 부여
> ```

### Q: 캐릭터 일관성(Character Consistency)은 어떻게 보장하나요?
> **A:** 이것이 이 프로젝트의 핵심 기술적 차별점입니다.
> 1. 사용자가 캐릭터 **참조 이미지(reference image)**를 업로드합니다.
> 2. Claude API에 캐릭터 이름과 외모 설명을 전달하여, 씬 프롬프트에 캐릭터 묘사를 포함시킵니다.
> 3. FLUX Kontext 모델에 `image_url` 파라미터로 참조 이미지를 전달합니다.
> 4. 이 파이프라인 덕분에 모든 장면에서 동일한 캐릭터 외모가 유지됩니다.

### Q: 프로젝트당 Plot이 1개만 가능한 이유는?
> **A:** 현재 비즈니스 요구사항에서 하나의 프로젝트는 하나의 스토리라인을 가집니다. 여러 Plot을 허용하면 UI/UX가 복잡해지고, 캐릭터 일관성 관리도 어려워집니다. 서비스 레이어에서 이미 Plot이 존재하면 409 Conflict를 반환하여 강제합니다. 추후 필요 시 쉽게 확장할 수 있습니다.

## 8. 성능/최적화 관련

### Q: 성능 최적화를 위해 어떤 노력을 했나요?
> **A:**
> 1. **DB 인덱스 최적화:** 조회 패턴에 맞춘 복합 인덱스 설계
> 2. **커넥션 풀 튜닝:** HikariCP 설정을 비동기 작업 특성에 맞게 조정
> 3. **Lazy Loading:** `@ManyToOne(fetch = LAZY)`로 N+1 문제 방지
> 4. **비동기 스레드 풀:** I/O 바운드 작업에 맞는 스레드 수 설정
> 5. **영상 해상도 최적화:** 생성 속도와 품질의 트레이드오프를 고려해 480p/20스텝 설정
> 6. **지수 백오프 폴링:** 외부 API 서버 부하 최소화

### Q: N+1 문제가 뭐고, 어떻게 해결했나요?
> **A:** N+1 문제는 부모 엔티티 1건 조회 후, 연관된 자식 엔티티를 건별로 N번 추가 쿼리하는 문제입니다. 예를 들어 10개 프로젝트를 조회하면 각 프로젝트의 Plot을 가져오기 위해 10번의 추가 쿼리가 발생합니다. 저희는 `FetchType.LAZY`로 기본 설정하고, 필요한 경우에만 연관 데이터를 접근하여 불필요한 쿼리를 방지했습니다.

## 9. 트러블슈팅 / 어려웠던 점

### Q: 개발 중 가장 어려웠던 문제는?
> **A:**
> 1. **Claude 응답 파싱:** Claude가 JSON 응답을 마크다운 코드블록으로 감싸거나, 추가 텍스트를 붙이는 경우가 있어 파싱 로직을 보강해야 했습니다. JSON 배열의 시작(`[`)과 끝(`]`)을 찾아 추출하는 로직을 구현했습니다.
> 2. **캐릭터 일관성:** 초기에는 text-to-image만 사용했는데, 같은 캐릭터가 장면마다 다르게 생성되는 문제가 있었습니다. FLUX Kontext의 참조 이미지 기능을 도입하여 해결했습니다.
> 3. **영상 생성 타임아웃:** FAL.ai 영상 생성이 예측 불가능한 시간이 걸려서, 지수 백오프 + 최대 타임아웃 조합으로 안정적으로 처리했습니다.
> 4. **이미지 포맷 호환성:** AVIF 등 FAL.ai가 지원하지 않는 이미지 포맷을 업로드하면 생성이 실패하는 문제가 있어, 업로드 시점에서 지원 포맷(JPEG, PNG, WebP, GIF)만 허용하도록 검증을 추가했습니다.

### Q: 만약 AI API 서비스가 다운되면 어떻게 되나요?
> **A:** 현재는 해당 생성 요청이 FAILED 상태로 처리되고, 사용자가 재시도할 수 있습니다. 개선 방향으로는:
> 1. **Circuit Breaker 패턴** 도입: 연속 실패 시 빠르게 에러를 반환하여 리소스 낭비 방지
> 2. **Fallback 모델:** FAL.ai 장애 시 대체 모델 사용
> 3. **알림 시스템:** 외부 API 에러율이 임계값을 넘으면 운영자에게 알림

## 10. 확장성 / 향후 계획 관련

### Q: 사용자가 많아지면 어떻게 대응하나요?
> **A:**
> 1. **수평 확장:** JWT Stateless 인증이라 서버를 여러 대로 늘릴 수 있습니다. 단, 비동기 작업 상태가 DB에 저장되므로 어느 서버에서든 상태 확인이 가능합니다.
> 2. **메시지 큐 도입:** 현재 인메모리 스레드 풀 대신 Redis/RabbitMQ를 도입하면, 작업 큐를 별도 워커 서버에서 처리할 수 있습니다.
> 3. **캐싱:** 갤러리 목록, 인기 콘텐츠 등 읽기 빈도가 높은 데이터에 Redis 캐시 적용
> 4. **CDN:** 생성된 이미지/영상을 CDN으로 배포하여 응답 속도 개선

### Q: 이 프로젝트에서 개선하고 싶은 점은?
> **A:**
> 1. **WebSocket/SSE 도입:** 폴링 대신 실시간 상태 알림으로 사용자 경험 개선
> 2. **테스트 코드:** 단위 테스트와 통합 테스트를 추가하여 안정성 강화
> 3. **모니터링:** Spring Actuator + Grafana로 AI API 응답시간, 에러율 모니터링
> 4. **비용 최적화:** 생성 결과 캐싱, 중복 요청 방지 등으로 API 호출 비용 절감
> 5. **프롬프트 최적화:** A/B 테스트로 더 나은 프롬프트를 찾아 생성 품질 향상

## 11. 기술 면접 빈출 개념 (이 프로젝트 맥락)

### Q: REST vs GraphQL, 왜 REST를 선택했나요?
> **A:** 현재 API 구조가 CRUD 기반으로 단순하고, 리소스 간 관계가 깊지 않아 REST로 충분합니다. GraphQL은 복잡한 중첩 쿼리가 필요할 때 유리하지만, 학습 곡선과 캐싱 복잡성이 더 큽니다.

### Q: JPA의 영속성 컨텍스트(Persistence Context)란?
> **A:** JPA가 관리하는 엔티티의 1차 캐시입니다. 같은 트랜잭션 내에서 같은 ID로 조회하면 DB에 다시 쿼리하지 않고 캐시에서 반환합니다. 또한 트랜잭션 커밋 시 변경 감지(Dirty Checking)로 자동으로 UPDATE 쿼리를 생성합니다. 저희 프로젝트에서 이미지/영상 상태를 PROCESSING → COMPLETED로 변경할 때 이 메커니즘이 작동합니다.

### Q: Spring의 @Transactional은 어떻게 동작하나요?
> **A:** AOP(Aspect-Oriented Programming) 기반 프록시가 메서드 시작 시 트랜잭션을 열고, 정상 종료 시 커밋, 예외 발생 시 롤백합니다. 주의할 점은 같은 클래스 내부에서 호출하면 프록시를 거치지 않아 트랜잭션이 적용되지 않는다는 것입니다. 저희 프로젝트에서는 서비스 레이어에 `@Transactional`을 적용하고, 비동기 메서드는 별도 빈(FalAiService)에서 호출하여 트랜잭션이 정상 동작하도록 했습니다.

### Q: Spring WebClient vs RestTemplate 차이는?
> **A:** RestTemplate은 동기(blocking) 방식이고 Spring 5부터 유지보수 모드입니다. WebClient는 비동기/논블로킹을 지원하며, `.block()`을 사용하면 동기처럼도 쓸 수 있습니다. 저희 프로젝트에서는 Claude API 호출은 `.block()`으로 동기 처리하고, FAL.ai 영상 폴링은 WebClient의 반응형 체인을 활용합니다.
